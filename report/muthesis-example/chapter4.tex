\chapter{System Design and Implementation}
\label{cha:system-design-implementation}

\section{System Overview}
\label{sec:system-overview}

This chapter describes Polaris, a retrieval-augmented assistant designed for supervised \ac{CSD3} support question answering. It implements the design requirements defined in Chapter~\ref{cha:problem} as a modular retrieval-augmented support system. Its architecture separates offline evidence preparation from runtime query handling so that corpus construction, metadata enrichment, retrieval, generation, and auditability can be controlled independently. The system indexes local documentation, external official documentation, and historical support tickets into source distinct stores, with authority and validity metadata used to preserve differences in reliability and operational relevance. This separation is central to Polaris's design, allowing runtime retrieval to combine heterogeneous evidence while still exposing the metadata needed for reviewable support responses.

Offline construction ingested the evidence sources, converted them into a consistent representation, enriched them with metadata, and indexed them into source-distinct Qdrant collections together with per-source document stores \citep{CollectionsQdrant}. At runtime, the FastAPI service coordinated registry-backed query interpretation, multi-collection retrieval, reranking, context resolution, prompt construction, LLM generation, and response packing. The observability layer recorded traces, prompt versions, feedback, and run artefacts, supporting reproducibility and making system behaviour auditable during evaluation and debugging.

Figure~\ref{fig:polaris-query-flow} illustrates how an individual user query moves through the system. A query is first validated by the API and associated with runtime tracing information. The system then extracts registry-backed constraints, such as software, version, scope, and intent signals, before retrieving evidence from the configured source-distinct stores. Retrieved candidates are then fused and ranked using \ac{RRF}, with validity signals available where enabled. The selected evidence is resolved into final contexts, rendered into the generation prompt, and passed to the LLM. The generated answer is then packaged by trimming output drift, deriving an evidence status from the retrieved context count, and logging the relevant outputs for auditability.  

The remainder of this chapter decomposes this overview into the main implementation decisions and design trade-offs made during Polaris's development. 

\section{Baseline Systems}
\label{sec:baseline-systems}

This project implements a series of baseline systems to evaluate the impact of specific design choices in Polaris. Each baseline system is a controlled comparison version of the main Polaris system, with specific design choices removed to isolate their effects. The generator, API pipeline, prompt format, and evaluation protocol are held constant across all systems to ensure that any observed differences in performance can be attributed to the design choices being evaluated.

Two baseline systems were implemented for \hyperref[rq:1-source]{Research Question 1}. The first, \texttt{docs\_only}, retrieved only from the local \ac{CSD3} documentation collection and measured how far local documentation alone could support the benchmark queries. The second, \texttt{tickets\_only}, retrieved only from the historical support ticket collection and measured how far historical resolutions alone could support the same query set. Comparing these baselines with the combined system allowed the evaluation to test whether heterogeneous retrieval improved answer quality for the benchmark query mix, or whether one source dominated once the generator, prompt, and API pipeline were held constant. 

A single baseline system was implemented for \hyperref[rq:2-validity]{Research Question 2}. This baseline, \texttt{naive\_combined}, used both local documentation and historical tickets with standard \ac{RRF} retrieval but without metadata-driven reranking. It therefore acted as the direct implementation comparator for the proposed validity-aware reranking design. Additional retrieval variants were also implemented as ablations, but were not treated as formal baselines.  

\section{Corpus Ingestion and Representation}
\label{sec:corpus-ingestion-representation}

This section describes how raw evidence sources were ingested into retrieval-ready evidence units. Local \ac{CSD3} documentation, external documentation, and resolved support tickets were ingested through separate pipelines, but source boundaries were preserved so that later source-aware retrieval and reranking remained explicit.

In this dissertation, chunking referred to splitting a longer source document into smaller bounded text units that could be independently embedded, indexed, and retrieved. This choice mattered because whole-document retrieval was too coarse, while overly small chunks risked losing important context \citep{wangDocumentSegmentationMatters2025}. 

For the documentation sources, the ingestion pipeline first preprocessed the HTML pages and then converted them into a markdown-normalised representation before token-based chunking. Markdown conversion was used because the source material originated from heterogeneous HTML structures, whereas markdown preserved useful structure such as headings, lists, code blocks, and links in a consistent text format while removing much of the presentation-specific noise of raw HTML. The final documentation setting used 400-token chunks with no overlap. The same setting was used for the external documentation source. 

For historical support tickets, the ingestion pipeline first queried the Atlassian Jira API for resolved tickets, converted the JSON content into a structured transcript format, and then applied a dialogue-based chunking strategy. A dialogue-based representation was used because important support evidence often spans short sequences of messages rather than a single isolated passage, and prior work has shown that historical tickets preserve useful issue-resolution context for support workflows \citep{rodenbeckTicketHubEnablingActionable2023,joslinGeneratingFrequentlyAsked2025}. Each ticket was represented using its key, summary, status, available timestamps, initial description, and subsequent conversational exchange. Speaker roles were derived from Jira user fields and preserved at the message level. Ticket-level status and timing information were retained as metadata rather than embedded directly into every chunk. In the final system, ticket chunks used a $1000$-token budget, $80$ tokens of overlap, and a maximum of three distinct turns per chunk. Retrieved ticket chunks were then used to identify relevant tickets, after which the corresponding full ticket transcript was resolved for generation so that retrieval remained fine-grained while answer generation included the broader problem-resolution context. 

The final chunking settings were selected empirically on the development split. Documentation-only tuning selected $400$ tokens with zero overlap because it gave the highest mean factual correctness, $0.299$, among the tested documentation settings. Ticket-only tuning selected $1000$ tokens with $80$ tokens of overlap because it gave the highest mean factual correctness, $0.326$. A combined follow-up then retained the 400/0 plus 1000/80 pairing, which produced the highest mean factual correctness, $0.329$, of the tested mixed settings. Section~\ref{sec:chunking-tuning} reports the full comparison. 

\section{Authority Registry and Metadata Extraction}
\label{sec:authority-registry-metadata-extraction}

Once the source documents have been converted into retrieval-ready representations, the system requires a structured authority layer to identify the technical entities they mention and to attach the metadata needed for later retrieval decisions. This is implemented as an offline authority registry that normalises systems, partitions, services, software packages, modules, and toolchains together with their associated scope, version, and lifecycle information. During ingestion, this registry is used to enrich source records before their metadata is propagated to the retrieval chunks. 

To ensure deterministic and auditable metadata enrichment, the authority registry is built offline, prior to runtime query handling. Local \ac{CSD3} documentation, and where configured, registered external documentation sources are first converted into separate markdown-normalised corpora. These corpora are scanned for candidate entities, which are then merged into normalised registry entries so that repeated mentions across pages map to a single stable definition. Ambiguous aliases or conflicting lifecycle statuses are not resolved silently. Instead, they are added to a review queue for manual inspection. The resulting artefacts are persisted as stable JSON along with build metadata such as source scope and extraction version, supporting traceability and reproducibility. For the documentation sources, metadata extraction is mainly done by matching the document's source identity against registry entries so that each chunk inherits the entity information associated with its originating document. On the other hand, for the ticket sources, metadata extraction is text-driven: ticket content is scanned using alias matching, \texttt{module load} parsing, and version mention extraction. The resulting matches are then resolved against the same authority registry. This process attaches authority, entity, scope, validity, freshness, provenance, and governance signals to each retrieval unit, with the complete field-level schema summarised in Appendix~\ref{app:authority-metadata-fields}. Using the same registry for both documentation and ticket metadata extraction ensures that they are both interpreted through one consistent set of entity definitions, rather than source-specific rules. 

This metadata is not stored solely for description, but also to support later runtime decisions. Since each retrieval unit carries explicit authority, scope, version, validity, and provenance signals, the same registry-backed representation can be used for query constraint parsing, source-aware retrieval, and validity-aware reranking. The authority layer therefore acts as a bridge between offline corpus preparation and online answer generation. Furthermore, the authority layer also improves auditability, since retrieved evidence can be traced back to its original source, matched entities, and extraction context, making errors easier to inspect during evaluation and debugging. 

\section{Validity-Aware Reranking Design}
\label{sec:validity-aware-reranking-design}

As discussed in Chapter~\ref{cha:background}, semantically relevant evidence could still be unsuitable if it was less authoritative, outdated, or locally inapplicable \citep{liYouKnowWhat2024,hwangRetrievalAugmentedGenerationEstimation2025}. The system therefore did not treat retrieval as a purely relevance-based problem over a flattened homogeneous corpus. Instead, initial semantic retrieval was followed by a metadata-driven reranking stage that could promote evidence that was more authoritative, current, and locally applicable. This design was implemented as a testable hypothesis rather than being assumed to improve the results prior to evaluation. 

After query constraints had been derived, the system retrieved candidates using a shared hybrid budget of eight dense retrievals and eight BM25 retrievals per source. Per-source rankings were fused using Reciprocal Rank Fusion (\ac{RRF}) with $k=60$, preserving source identify for later tracing, and the fused candidate set was truncated to a final top $8$ before context resolution \citep{cormackReciprocalRankFusion2009}. The semantic relevance-only fused ranking corresponded to the \texttt{naive\_combined} baseline introduced in Section~\ref{sec:baseline-systems} and served as the direct implementation comparator for the reranking design. 

As illustrated in Figure~\ref{fig:polaris-reranking}, the validity-aware reranker began with the normalised semantic base score derived from the \ac{RRF} ordering. The final reranking score was formed by adding weighted metadata features to this semantic base. In the final experimentally tuned configuration, authority and exact scope match each received weight $0.08$, lifecycle status and version each received $0.04$, scope-family received $0.02$, and relative freshness received $0.01$. Software match was implemented by received zero weight in the selected configuration. Authority values were mapped as $1.0$ for local \ac{CSD3} documentation, $0.5$ for external documentation, and $0.0$ for historical tickets. Lifecycle values were mapped as $1.0$ for current, $0.25$ for maintenance, $-0.5$ for legacy, and $-1.0$ for end-of-life.

    \caption[Validity-aware reranking in Polaris.]{Validity-aware reranking in Polaris. Source-distinct candidates are merged and fused into an RRF semantic base, then adjusted using query-aware validity features derived from registry-backed constraints, enriched metadata, and the configured reranker profile before final evidence selection.}
    \label{fig:polaris-reranking}
\end{figure}

These validity signals were not applied blindly to every query. The reranker used the derived query constraints to control how individual metadata features behaved. This was especially important for software-constrained queries, where a broad scope-family match could otherwise promote evidence that referred to the correct platform family but not the relevant software package. In such cases, scope-family boosts were gated unless the candidate also satisfied the required software match, and specialised family variants were prevented from receiving an inappropriate generic boost when the exact variant had not actually been requested. On the other hand, for non-software operational queries, scope-family could still provide a useful positive signal. 

The final reranker profile was selected on the development split by a two-phase grid search over $47$ trials using a coverage-guarded selection rule based on factual-correctness, faithfulness, and context precision. The retained profile achieved development means of $0.314$ for factual correctness, $0.613$ for faithfulness, and $0.739$ for context precision. The active reranker configuration was persisted as a structured profile and fingerprint and logged alongside retrieval traces and final outputs so that ranking behaviour could be reconstructed across runs. Section~\ref{sec:validity-reranker-tuning} describes the full protocol in full. 

\section{Prompt and Generation Design}
\label{sec:prompt-generation-design}

The generation stage was designed to produce grounded, reviewable support responses. A single fixed prompt template was used within each retrieval comparison so that measured differences could be attributed to retrieval and reranking rather than prompt drift. The final prompt retained a ticket-aligned response schema with fixed sections for classification, assessment, action, customer reply, and safety notes, together with inline numeric citations and a \texttt{REFERENCE KEY}. Two few-shot demonstrations were included: one evidence-sufficient case and one insufficient-evidence case. These demonstrations established the target response structure and modelled conservative behaviour when retrieved evidence was weak \citep{brownLanguageModelsAre2020,asaiSelfRAGLearningRetrieve2023,gaoEnablingLargeLanguage2023}.

For \ac{HPC} support, incorrect operational guidance was treated as more harmful than an incomplete answer. When evidence was missing or conflicting, the prompt therefore required explicit uncertainty, the safest next step, and concise follow-up questions rather than unsupported resolution. After generation, post-processing trimmed output drift and rebuilt the \texttt{REFERENCE KEY} from citations actually used in the answer body, preventing uncited reference mappings from appearing in the final output. 

A later follow-up prompt ablation was conducted after the main error analysis identified generation, rather than retrieval alone, as the remaining performance bottleneck. The refined prompt variant is therefore reported as an additional improvement study rather than as part of the frozen retrieval backbone. The generator configuration used \texttt{meta-llama/Llama-3.3-70B-Instruct}, selected in the development experiments, with a $2048$-token output limit, temperature $0.8$, and top-p $0.9$.

\section{API, Tooling, and Observability}
\label{sec:api-tooling-observability}

FastAPI was used to expose the system as a typed HTTP service because it provides structured requests and response validation and a stable interface for both the interactive UI and the evaluation tooling \citep{FastAPI}. The main API endpoint, \texttt{/v1/query}, returns a structured response containing the generated answer, the retrieved contexts, derived query constraints, answer-status metadata, and timing metadata. 

The FastAPI requests are governed by a shared request-budget model that allocates a single end-to-end deadline across retrieval and generation. Stage-specific limits are derived from the remaining budget, with an explicit retrieval cap and reserved cleanup margin. This prevents retrieval from exhausting the request window before answer generation completes. The same API also supports optional evaluation metadata, while timeout and internal failures are returned as structured HTTP payloads for downstream inspection. 

Operational tooling is exposed through lightweight service endpoints. Basic liveness is separated from dependency-aware readiness by providing both a simple \texttt{/health} endpoint and a more complex \texttt{/ready} endpoint that checks local runtime dependencies such as the embedding service and the required Qdrant collections. The API also records and aggregates structured user feedback. 

The system also records MLflow trace spans across both API and pipeline stages, with correlation headers linking evaluation requests to runtime traces \citep{MLflowOpenSource}. These traces capture prompts, retrieved contexts, query constraints, timings, and retriever or reranker profiles, making failures easier to debug and experimental runs easier to audit. 

\section{User Interface and Interaction Design}
\label{sec:user-interface-interaction-design}

The frontend was implemented as a React single-page application with separate assistant, evaluation, and system views. This separation was chosen so that interactive querying, structured scenario evaluation, and runtime inspection could support different tasks without overloading a single screen. The assistant view opens with both free-text input and predefined scenario prompt cards, allowing either open-ended use or rapid demonstration of representative support queries. After submission, the workspace presents the latest answer alongside a diagnostics panel rather than mixing evidence with the main response. This split keeps the primary support answer readable while still allowing timings, interpreted query constraints, retrieved evidence chunks, and optional debug metadata to be inspected when needed. 

As shown in Figure~\ref{fig:polaris-ui}, the evaluation view reuses the same answer and diagnostics components but wraps them in predefined scenarios and a structured feedback form. This allows scenario runs to be compared under a consistent interface while capturing helpfulness, groundedness, citation quality, failure type, and free-text notes. Response fingerprinting is used to prevent duplicate in-session submissions for the same output, making frontend feedback directly reusable in later analysis. A separate system view exposes health, readiness, and runtime configuration so that operational state can be inspected without interfering with either querying or evaluation. 

\section{Verification and Software Testing}
\label{sec:verification-testing}

This section describes the verification strategy used to check the implementation prior to formal evaluation. The codebase contained $394$ named pytest test functions across $53$ test modules. The suite was organised by verification layer rather than by package alone so that the highest-risk behaviours could be isolated and regression-tested before benchmark execution. Verification was risk-driven, with priority given to behaviours most likely to affect answer correctness, runtime robustness, and reviewability. At the component level, tests target the project's highest-risk logic directly. Authority-registry tests checked deterministic artifact construction, candidate extraction, and stable metadata enrichment. Query-constraint tests checked alias resolution, scope-family inference, and version-sensitive parsing. Ticket-chunking tests verified token-budget enforcement, overlap handling, and the three-turn chunk limit. Validity-reranker tests verified official-match promotion, software-gated scope-family boosts, and rerank trace stamping. Pipeline and API tests checked answer-status and timing payloads, reference-key sanitisation, evaluation metadata handling, and the mapping of timeout failures to structured 504 responses. Additional operational tests covered readiness checks, tracing behaviour, prompt-registry integration, and evaluation-facing API paths. This suite did not establish answer quality, which was handled separately by the held-out benchmark, but it did reduce the risk that later empirical findings were artefacts of broken retrieval, prompting, or API plumbing. 

\section{Design Trade-offs}

The main drawback of this design was that it introduced a large number of hand-tuned controls. Preserving source distinctions, maintaining the authority registry, tuning chunk settings, and tuning reranker weights improved auditability, but they also introduced maintenance burden and additional failure modes. Stale registry entries could lead to incorrect metadata enrichment and the hand-tuned reranker weights could overfit the development split. For this reason, these controls were treated as empirical design choices to be tested later rather than as assumptions of automatic improvement. 

A similar trade-off appeared at runtime. Structured prompts, reference keys, diagnostics, tracing, and request-budget controls reduced flexibility and added overhead compared with a lightweight question-answering interface. However, these constraints made answers easier to inspect, debug, and evaluate, which was more important for this project than maximising conversational freedom. This trade-off was justified for a supervised support assistant, but it also meant that development effort was spent on control, auditability, and evaluation support rather than on a simpler end-user experience.
